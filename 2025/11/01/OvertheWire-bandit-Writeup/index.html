OverTheWire-Bandit的通关经历

谨以此纪念本人在学习Linux命令行过程中通过bandit游戏辅助学习的经历

Level0:建立ssh连接

本题要求通过2220端口，建立ssh连接进入bandit.labs.overthewire.org，且提示账密均为bandit0，那么我们使用命令

ssh -p 2220 bandit0@bandit.labs.overthewire.org

然后在弹出输入密码的要求的时候输入bandit0为密码即可进入系统

Level1:读取readme

本题称在readme里有进入下一关账号bandit1的密码，并且readme文件在home directory里

先ls一下确定根目录下有readme文件，然后再cat readme就可以得到下一关密码了

Level2:读取-

本题称密码在文件名为“-”的文件中，并且-文件在home directory里

先ls确认文件存在，但是这回因为“-”同时也是根目录的名字，所以不能直接cat，要用指定当前路径的cat方式

cat ./-

如此得到密码进入下一关

Level3:读取名字里带有空格的文件

本题称密码在一个叫“–spaces in this filename–”的文件，且其在home directory里

一样先ls,然后空格不能直接打要用转义字符 \ 打出，然后同时因为有-也要用默认路径的方式

cat ./--spaces\ in\ this\ filename--

如此得到密码进入下一关

Level4:读取隐藏目录里的文件

本题称密码文件在inhere目录里，那我们先cd inhere，但是ls却没发现任何的内容，猜测是在隐藏目录里头

所以我们用ls -a，看到了有个Hiding-From-You文件在三级子目录内，于是可以cat …Hiding-From-You得到密码

Level5:读取可读文件

本题称文件在inhere目录里，那我们先cd,然后ls发现有10个，其中有些里面是乱码

于是我们一个个cat，如果是乱码就按照题目提示reset，这样下去看到file07时得到密码

优化 可以先file ./*，查看文件格式，发现只有file07是txt其余都是dat（不可读），就可以直接cat它得到密码

Level6：寻找特征文件

本题称密码在一个可读的、1033bytes大小的文件里，所以我们用find命令整合

find . -type f -size 1033c -ls

可以遍历所有文件找到目标在maybehere07文件夹，但是ls发现夹里没有目标文件，加上它叫.file2，所以是隐藏

我们cat .file2即可找到密码

Level7:读取属于本组但不属于本用户所有的文件

本题称密码在一个大小33字节的文件内，该文件属于user:bandit7（下一关账户）所有，而也属于group:bandit6（本关组）所有

学习后得知，对于这个文件的权限，由于不属于当下用户所有所以我们没有编辑权限，但是由于是本组所有，所以我们有读取权限

所以我们用find命令查找user是bandit6，group是bandit7，且大小为33c的文件：

find -user bandit7 -group bandit6 -size 33c

在一大串如下的权限失败的提示里:

find: ‘/root’: Permission denied

find: ‘/etc/ssl/private’: Permission denied

find: ‘/home/bob’: Permission denied

/some/path/target_file

find: ‘/var/lib/mysql’: Permission denied

...

可以找到一条长这样的：
/var/lib/dpkg/info/bandit7.password

而根据上文，咱有这个文件的读取权限，直接cat就可以得到密码

优化解法

前置知识1 Linux黑洞：/dev/null，任何写入这里的数据都会丢失

前置知识2 Linux的3种标准数据流：0-stdin-标准键盘输入；1-stdout-标准输出；2-stderr-标准错误（错误信息）

那么知道这两个后，我们就可以想，怎么可以省去那一大串拒绝访问的错误提示，直接显示我们可以访问的那个路径呢？

哦豁，把他们写入数据黑洞里不就行了！

这下子衍生出了一条优化代码：

find / -user bandit7 -group bandit6 -size 33c 2>/dev/null

这样子所有的报错全部都写入黑洞丢失，当然就只剩下有用的内容了！

Level8:查找文本关键字

本题称密码在data.txt里的millionth这个关键字旁边，要我们去寻找

首先ls确认txt存在，然后首先尝试cat，但是发现data.txt里有成千上万行，压根肉眼找不过来

那我们只能用文本查找命令了，也就是grep

grep -C 3 "millionth" data.txt

这样子就是查找这个关键字所在行并显示前后三行（-A显示后几行，-B显示前几行）

millionth dfwvzFQi4mU0wfNbFOe9RoWskMLg7eEc

这里就是下一个账户的密码了

Level9:查找文本中只出现过一次的独特行

本题称密码在data.txt中出现过一次的行里，同理ls确认，然后cat肯定也行不通，这样的搜索要用sort

根据对sort的学习，可以用sort+uniq的组合命令略去全部重复过的行只留下仅出现一次的独特行，具体命令如下：

sort data.txt | uniq -u

这样就可以得到独特行，从而得到答案了

注意 sort -u也是可以删去重复行的，但它会为每种行留下一次输出，也就是输出每个类别，与此题要求不符合，还是要uniq才能得到符合的独特行

Level10:寻找由几个等号引导的特殊行

本题称密码在data.txt里，跟在若干等号后头，其实这题文本量不大可以直接cat找一下就找到了（bushi），但是其实这样子找到的密码多了字母，不对

看下更预期的解：首先我们尝试用grep寻找“=“，因为题目说a few “=”s所以应当使用正则表达式”={2,}”命令如下：

grep "={2,}" data.txt

结果报错了，说是grep: data.txt: binary file matches，好的，二进制文件是吧，我们还是强硬地用-a将其当成文本文件处理！

grep -a -E "={2,}" data.txt

然后可以找到所有连着等号的字段，分别等号后面跟着的是the password is FGUW5ilLVJrxX9kMYMmlN4MgbpfMiqey，得到密码

Level11:解码含有base64的txt

本题称密码在data.txt里且被base64加密，那么我们先进入系统后ls确认文件存在，然后cat看一下，里面只有一个一行的base64编码

那么我们就运用自带的命令解码base64即可，后面 | cat方便直接查看

base64 -d data.txt | cat

于是得到这样的输出：The password is dtR173fZKb0RRsDFSGsg2RWnpNVj3qRr，就可以知道密码了

Level12:解码含有Rot13的txt

本题称密码在data.txt里且被ROT13加密过，那么我们先进入系统后ls确认文件存在，然后cat查看，里面有一串Rot13加密过的字符

没有直接的命令可以解码ROT13，但是我们可以通过直接通过映射变换字符的命令tr，依据rot13的原理来解码

cat data.txt | tr'A-Za-z' 'N-ZA-Mn-za-m'

这就是把大小写字母映射到后13位的位置，就可以解密出文本为The password is 7x16WNeHIi5YkIhWsfFIqoognUTyj9Q4，得到密码

Level13:读取十六进制文件+多重解压

本题称密码在data.txt所存hexdump映射的文件中且这个文件被重重压缩，还建议我们去建立一个临时目录来进行读取和解压操作

那我们先建立临时目录：

mktemp -d

就可以在/tmp下建立一个临时目录，这样我们把data.txt拷贝过来，注意因为它本身是个hexdump了那我们去掉后缀让hexdump直接作为文件形式

cp ./data.txt /tmp/tmp.Lwm7yp1bgT/data

然后用xxd读取这个data文件，并转到名为compressed的文件中

xxd -r data > compressed

好，那么得到了一个未知格式的压缩包文件，用file查看格式

file compressed

发现是gzip文件，mv重命名添加后缀，解压然后再file……（重复多次操作） ->以下贴上本次三种压缩包的解压方式

gzip/bzip2 -d xxx.gz/xxx.bz2

tar -xf xxx.tar

注意 tar文件不是套壳型，所以解压不会修改原包，而是会把文件搞到这个目录下，但gz和bz2都是套壳也就是解压完了直接把源文件改掉了的

解压了gz,bz2,tar各三层后file看到这下得到的data8是个ASCII text，于是就可以cat得到密码

Level14:ssh的密钥连接通道

本题称密码在一个只有bandit14用户可读的文件，但是我们目前是bandit13用户，得不到密码，不过在bandit13的home路径里有个sshkey.private

这个就是ssh密钥，经学习了解到ssh命令除了直接连接、输入密码进入用户，还可以通过-i命令走密钥的通道进入用户

那么我们试试在bandit13里用这个文件登录

ssh -i sshkey.private bandit14@localhost -p 2220

但是却报错了，显示我们没有权限登录

在bandit13里既然没有权限登录，试试本地。那我们用scp命令

scp -P 2220 bandit13@bandit.labs.overthewire.org:sshkey.private ./bandit14_key

注意，这里如果不修改源文件名，就无法获得使用、读写该文件的权限（.private）

接下来我们修改读取权限为-rw——-（chmod 600 bandit14_key）

然后再用这个密钥按同上的方法登录即可

Level15:向另一个接口传输文本

本题称密码可以通过向localhost的30000接口传输上一关得到的密码来获得

那我们可以使用telnet命令在bandit14用户里连到localhost的30000接口，再输入密码即可

telnet localhost 30000

在终端输入密码就可以得到返回的本关密码答案

注意 本题也可以用nc进行连接，方法同西电ctf终端连题目的方式

Level16:向另一个接口通过ssl/tls协议传输文本

本题称密码可以通过使用ssl/tls协议向localhost的30001接口传输上一关得到的密码来获得

那我们这时候应该使用openssl命令来连接到主机

openssl s_client -connect localhost:30001

在终端输入密码就可以得到返回的本关密码答案

Level17:找到ssl/tls协议接受密码的接口来传输文本

本题称得到密码的方式与Level16一样，但是端口要从31000~32000里找到正确的端口，那么我们首先扫描哪些端口开放

nmap --open -p 31000-32000 localhost

找到了五个开放端口，31046，31518，31691，31790，31960，然后我们再依次找这些端口的证书

nmap -sV --script ssl-cert -p (端口名) localhost

看到其他都是echo返回输入值，只有31790是unknown，就是它！

同第十六关方法可以得到ssh密钥，用密钥登录下一关的用户！

Level18：俩文本找不同

本题称密码是文本passwords.new中与passwords.old唯一不同的那行

两种方法(diff/grep)

grep -v -f passwords.new passwords.old

diff -u passwords.new passwords.old

找出不同行，即可得到密码

Level19：绕过.bashrc强制断开的限制

本题称密码在home路径的readme里，但这个账户的.bashrc被修改，登陆后会强制登退

经学习我们知道，.bashrc只在登录交互式shell会被启动，非交互式的shell不会

那我们就可以直接用ssh执行命令，就能绕过.bashrc限制了

ssh -p 2220 bandit18@bandit.labs.overthewire.org "cat readme"

如此得到密码

Level20:setuid文件的用法

本题在home路径给出了一个setuid文件，并告诉我们密码在跟第十四关一样的位置

经学习setuid文件是可以让我们以另一个user的身份来执行命令，而通过

./bandit20-do whoami

可以得知这个user就是bandit20，那我们就

./bandit20-do cat /etc/bandit_pass/bandit20

就能得到密码了

Level21:setuid文件的用法2（suconnect）

本题在home路径给出了一个suconnect文件，经检查它是一个可以接收localhost的任意端口返还的字符的文件

收到后会与上一关密码检测是否匹配，然后配对成功就发送下一关密码

但是端口一开始没东西，所以我们要使用nc传输密码到一个对应端口来达到题目返还本关答案的要求

echo "0qXahG8ZjOVMN9Ghs7iOWsCfZyXOUbYO" | nc -l -p 30010

这样把密码传到30010端口，再./suconnect 30010就可以拿到下一关密码了

Level22:crontab时间表&.sh文件

本题提示我们去看看/etc/cron.d里的关于bandit22的文件，我们cd进去ls看到有个cronjob_bandit22

cat以后发现这个bandit22做了启动/usr/bin/cronjob_bandit22.sh这个操作，于是我们cat一下这文件看看究竟

这文件里又体现了它chmod 644 /tmp/t7O6lds9S0RqQh9aMcz6ShpAoZKF7fgv并cat /etc/bandit_pass/bandit22 > /tmp/t7O6lds9S0RqQh9aMcz6ShpAoZKF7fgv

也就是说把这个熟悉的存着密码的路径里的内容复制到了这个/tmp下这个文件里，我们cat这个文件就可以得到密码了

Level23:crontab,.sh和小计算

本题提示我们去看看/etc/cron.d里的关于bandit23的文件，我们cd进去ls看到有个cronjob_bandit23

cat以后发现这个bandit23做了启动/usr/bin/cronjob_bandit23.sh这个操作，于是我们cat一下这文件看看究竟

这里文件体现了它运用了usrname进行了计算，具体为echo I am user bandit23 | md5sum | cut -d ‘ ‘ -f 1

输入此命令得到转换后的字符串，cd /tmp/（该字符串）得到密码

Level26:另类的shell以及more查看器

本题称bandit26的shell不是/bin/bash而是另一个东西，并给了我们下一关的ssh密钥，让我们去找找看

先试试密钥，但是它是登上去立刻退了出来，而且用前面绕过.bashrc修改的方法尝试绕过也失败了

那我们先找找这个bandit26是怎么打开的，我们进入/etc/passwd，然后看到bandit26的shell是/usr/bin/showtext

我们cat到这个文本里，看到如下内容

那我们有没有办法不让exit 0执行呢？答案是有的

据学习可知 more查看器有两种可能会自动关掉：1、文件太小/屏太大不需要分页，2、文件为空

咱明知了文件不为空，那我们让它需要分页more就会接着执行不执行到下一步exit 0，就不会弹出来了，所以我们把窗口缩到最小再密钥连接

如此我们就可以进入到More查看器了，接下来，我们用vim语法，打出：

:r /etc/bandit_pass/bandit26

得到了bandit26的密码

Level26:用more打开shell

这道题延续上一题，提示我们bandit26的home路径里有东西要我们找回shell来把这个东西得到以获取下一关密码

那我们依旧采用上一关的方法卡出more，然后用vim语法进行set shell操作

:set shell sh=/bin/sh

:sh

于是就可以打开shell，ls看到home下面有一个bandit27-do，file一下是一个setuid文件，让我们可以以bandit27身份运行命令

./bandit27-do cat /etc/bandit_pass/bandit27

就可以得到bandit27的密码了

Level27:git初级使用

本题称密码存储在ssh://bandit27-git@bandit.labs.overthewire.org:2220/home/bandit27-git/repo目录下，密码就是上题得到的

所以我们用git clone命令：

git clone ssh://bandit27-git@bandit.labs.overthewire.org:2220/home/bandit27-git/repo

输入密码。然后里面的内容就复制到了本地home路径的repo文件夹里，cd进去看到README,cat一下得到密码

Level28:git查询版本历史

本题称密码存储在ssh://bandit28-git@bandit.labs.overthewire.org:2220/home/bandit28-git/repo目录下，密码就是上题得到的

所以我们用git clone命令：

这显然是错误的密码，说明密码可能被修改过，于是我们可以采用git查看版本历史的命令：

这是修改痕迹，那么上面那个就是咱要的密码了

Level29:git的分支查询

本题称密码存储在ssh://bandit28-git@bandit.labs.overthewire.org:2220/home/bandit28-git/repo目录下，密码就是上题得到的

所以我们用git clone命令：

git clone ssh://bandit28-git@bandit.labs.overthewire.org:2220/home/bandit28-git/repo

输入密码。然后里面的内容就复制到了本地home路径的repo文件夹里，cd进去看到README,cat一下，结果里面密码是<no passwords in production!>

照上一题的方法查看了版本历史，只发现了username从bandit30改成了bandit29，其他看不出什么变化，git log 查看日志也是一无所获……吗？尝试：

git log --oneline --graph --all

可以看到里面除了把usrname改动的操作，dev分支里也有修改过东西的痕迹，那我们先把分支切换到dev，并再次cat README.md就可以得到答案

git checkout dev

Level30:git标签隐藏秘密

本题称密码存储在ssh://bandit28-git@bandit.labs.overthewire.org:2220/home/bandit28-git/repo目录下，密码就是上题得到的

所以我们用git clone命令：

git clone ssh://bandit28-git@bandit.labs.overthewire.org:2220/home/bandit28-git/repo

输入密码。然后里面的内容就复制到了本地home路径的repo文件夹里，cd进去看到README,cat一下，结果里面说这是一个空文件哈哈哈

照上两题做法做了都没有得到有用的信息，不过我们可以使用命令搜索origin下的所有内容

git ls-remote origin

发现有个/tag/secret，这是个标签，估计藏了密码，那我们打开它就可以得到密码

git show secret

Level31:git传输文件到指定端口

本题中我们需要通过新建一个key.txt并将其push到一个远程的git仓库来获取下一关的密码，但是我们试了试直接这么做却报无权限（需密码不知道）

ls一下发现只有readme和key，但是ls -a发现了.gitignore这一个隐藏文件，cat一下里面显示了*.txt

这个文件做的事情是：git自动把txt后缀的文件忽略掉了，所以我们迟迟推不上去啊，那我们把这个ignore删掉后再新建一次文件就可以了

Level32:绕过Uppershell

本题让我们探索新的shell，我们登陆进来后他说WELCOME TO THE UPPERCASE SHELL

经过探索这个shell是必须大写命令，但是shell里啥也没有，就是套壳，我们得找到正常shell

经学习，我们有一种调取环境变量（$0）绕过的方法，原理就是输入这个会展开原始变量，相当于绕过upper的限制进入原始shell

那么我们接下来，就可以cat /etc/bandit_pass/bandit33得到密码了
